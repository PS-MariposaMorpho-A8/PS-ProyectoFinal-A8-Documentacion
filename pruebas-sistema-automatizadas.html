Fuera del alcance formal (Hito 3):
Por restricciones de tiempo, los siguientes grupos de pruebas quedaron en tests/archived/ y no se ejecutan como parte del entregable final: Log de Trazabilidad, Consentimiento Informado, Configuración y Control de Sesión.

Exclusión permanente: Unity/Kinect

El módulo de Unity/Kinect (lanzamiento del juego terapéutico, resultados de sesión con datos del sensor, generación de reportes PDF con métricas clínicas) no está desplegado en el entorno QA y el equipo no dispone del hardware Kinect para pruebas. En consecuencia:

  • Ningún escenario formal depende de datos Kinect reales.
  • Los casos que sí lo requerían (grupos RES y REP) quedan en tests/archived/ con guardas condicionales (test.skip).
  • Donde un flujo de usabilidad toca el borde de esta limitación (ej. sección "Evolución" sin sesiones previas), la prueba valida el estado vacío correcto de la interfaz.

Datos de prueba: usuarios efímeros vía API

La suite crea sus propios datos de prueba en cada ejecución:

  • global-setup.ts corre antes de la suite y crea un especialista de prueba vía POST /api/v1/users/ con email único (timestamp).
  • También realiza el cambio de contraseña obligatorio vía POST /api/v1/auth/change-password antes de que arranque cualquier test de UI, evitando el modal bloqueante de "primer ingreso".
  • Las credenciales finales se guardan en tests-system/.auth/specialist-credentials.json.
  • security.spec.ts crea sus propios especialistas y pacientes efímeros directamente vía API para los casos de control de acceso.

Configuración de Playwright

export default defineConfig({
  testDir: './tests',
  globalSetup: './global-setup.ts',
  timeout: 60_000,
  workers: 1,
  retries: process.env.CI ? 1 : 0,
  reporter: [['html', { open: 'never' }], ['list']],
  use: {
    baseURL: process.env.BASE_URL || 'https://rehab-frontend-qa.vercel.app',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  projects: [{ name: 'chromium', use: { ...devices['Desktop Chrome'] } }],
})

Nota sobre workers: 1: El entorno QA es un backend compartido y con estado. Ejecutar múltiples navegadores en paralelo contra él generó timeouts masivos. Correr en serie es más lento pero confiable.

Hallazgos Relevantes

Bug de aplicación: PS-USR-003 — endpoint incorrecto

Estado: Documentado y omitido (test.skip) con motivo explícito.
Descripción: Al editar los datos de un especialista, el frontend envía la petición a PATCH /api/v1/users/{id} (que exige new_password). El endpoint correcto es PATCH /api/v1/users/{id}/profile.
Conclusión: Un defecto real de integración frontend-backend fue detectado por la prueba automatizada, bloqueando la edición de usuarios desde la UI.

Diseño de security.spec.ts (Nivel API)

Las pruebas de seguridad se implementaron como llamadas HTTP directas contra la API, usando el request fixture de Playwright, generando datos efímeros:

  • SEC-001: petición GET /api/v1/patients/ sin token → se espera 401.
  • SEC-002: especialista intenta POST /api/v1/users/ → se espera 403 (requiere rol admin).
  • SEC-003: especialista intenta GET /api/v1/patients/{id} sobre paciente de otro especialista → se espera 403 (acceso denegado al recurso).

Automatización e Integración Continua

La suite está integrada al pipeline de CI del repositorio de backend (ci.yml, GitHub Actions) como el job system-tests.

  • Instala Node.js 20, dependencias y Chromium.
  • Ejecuta npm test usando credenciales como GitHub Secrets.
  • Sube el reporte HTML de Playwright como artifact (system-tests-report).
  • Depende del job unit-tests, ahorrando minutos de CI si las pruebas base fallan.
