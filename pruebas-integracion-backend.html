Plan de Pruebas de Integración

Implementación y ejecución del plan de pruebas de integración para el backend desarrollado en FastAPI. A diferencia de las pruebas unitarias basadas en mocks, estas pruebas interactúan directamente con una base de datos real (SQLite), verificando de extremo a extremo endpoints, reglas de negocio, seguridad (roles y permisos), triggers y vistas de base de datos, y los flujos integrados de pacientes, sesiones de terapia, documentos PDF y comunicación con Unity.

Casos PI-BACK

Fixtures de configuración (conftest.py)

Componentes reutilizables que preparan el entorno de cada caso de prueba.

db

Configura la sesión de base de datos para cada caso, envolviendo la ejecución en transacciones para mantener el aislamiento y revertir cambios de ser necesario.

client

Proporciona una instancia de TestClient de FastAPI configurada con la base de datos de integración y los overrides de dependencias adecuados.

admin_headers / specialist_headers

Encabezados HTTP con el token JWT Bearer pre-calculado para las credenciales por defecto (admin@local.test / specialist@local.test).

monkeypatch

Módulo de pytest utilizado para simular variables de configuración dinámicas y evitar el envío real de correos SMTP en la recuperación de contraseñas.

Conclusión técnica final

La suite de pruebas de integración desarrollada para el backend FastAPI cumple de forma rigurosa con los objetivos y criterios del plan de pruebas:

1. Robustez y veracidad

Al no apoyarse en mocks falsos para bases de datos, las pruebas aseguran que las sentencias SQL generadas por SQLAlchemy son plenamente compatibles con el motor SQLite y que todas las restricciones de integridad y triggers se aplican en la vida real.

2. Seguridad y aislamiento

Las verificaciones cruzadas entre Administradores y Especialistas confirman la efectividad de las directivas de seguridad basadas en roles, asegurando la privacidad de los datos demográficos e históricos de los pacientes.

3. Mantenibilidad a largo plazo

La parametrización de credenciales fijas y la configuración estructurada de base de datos temporal facilitan que estas pruebas puedan ejecutarse de forma automática en tuberías de CI/CD para evitar regresiones lógicas ante futuros cambios en el backend.
